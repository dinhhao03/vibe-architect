Chapter 1 Mô Tả Sản Phẩm

1.1 Tổng Quan (Product Overview)

Vibe-Architect là một AI-Powered Scaffolding Engine — hệ thống sinh cấu trúc dự án tự động chạy hoàn toàn trên môi trường Local (không phụ thuộc Cloud). Sản phẩm khai thác sức mạnh của mô hình ngôn ngữ lớn (LLM) Google Gemini thông qua kiến trúc điều phối đa tác tử (Multi-Agent Orchestration), một bước tiến vượt bậc so với phương pháp "hỏi một câu nhận tất cả" truyền thống vốn rất hay gây ra hiện tượng Hallucination (AI tự tưởng tượng sai lệch) trong các hệ thống AI thế hệ trước.

Từ một đoạn văn bản ý tưởng ngắn gọn (Prompt), Vibe-Architect lần lượt gọi 3 luồng AI chuyên biệt để phân tích nghiệp vụ, thiết kế cơ sở dữ liệu, và viết toàn bộ mã nguồn Backend; sau đó đóng gói tất cả thành một file .zip chuẩn Production-Ready có thể tải về và chạy ngay lập tức.

1.2 Bài Toán Được Giải Quyết

Trong quy trình phát triển phần mềm truyền thống, giai đoạn khởi tạo dự án (Scaffolding) tốn rất nhiều thời gian: lập trình viên phải tự tay viết tài liệu SRS, thiết kế ERD, tạo SQL Schema, dựng cấu trúc thư mục MVC và khởi tạo hàng chục file boilerplate. Với một dự án vừa, công việc này có thể mất 2–5 ngày làm việc.

💡 Giải pháp của Vibe-Architect

Tự động hoàn toàn giai đoạn Scaffolding bằng AI Pipeline chia lớp, đảm bảo nội dung nhất quán xuyên suốt từ SRS → ERD → SQL → Code Backend, thay vì phụ thuộc vào một phiên chat AI đơn thuần. Thời gian rút xuống còn dưới 30 giây.

2.2 Mô Hình 3-Agent Orchestration Pipeline

Đây là trái tim của toàn bộ hệ thống. Thay vì một lệnh gọi AI đơn thuần, mọi yêu cầu được xử lý qua chuỗi 3 não bộ chuyên biệt (Chain-of-Thought), mỗi Agent chỉ làm đúng nhiệm vụ của mình và truyền kết quả xuống Agent kế tiếp. Điều này loại bỏ hoàn toàn nguy cơ Hallucination.

1

🧠 Agent 1 — Business Analyst

Input: Prompt văn bản tự do của người dùng  |  Output: JSON gồm projectName, projectOverview, userStories[]

Phân tích ngữ nghĩa, tóm lược nghiệp vụ, sinh ra Danh Sách Yêu Cầu chuẩn (User Stories) với mã định danh ID.

↓
2

🏛️ Agent 2 — Data Architect

Input: projectName + userStories[] từ Agent 1  |  Output: JSON gồm erd (Mermaid), sql (PostgreSQL DDL)

Phân tích nghiệp vụ, ánh xạ sang mô hình dữ liệu chuẩn 3NF có PK/FK, xuất ra ERD và SQL DDL.

↓
3

💻 Agent 3 — Backend Developer

Input: projectName + sql từ Agent 2  |  Output: JSON gồm controllers[], services[], routes[], appCode, serverCode, dbCode, dependencies[]

Đọc SQL gốc để sinh code Controller/Route/Service khớp 100% tên Bảng. Tối đa 3 Entity cốt lõi cho MVP. Output dynamic array of files.

↓
✓

🛡️ Validation Layer + ZIP Packaging

Kiểm tra từng trường dữ liệu. Nếu thiếu/sai → chèn Fallback Template tĩnh. Sau đó ScaffoldFactory ghi file động lên RAM và pipe vào archiver ZIP.

2.3 Cơ Chế Truyền Context (Context Locking)

Vấn đề: Nếu mỗi Agent được gọi độc lập, AI có thể tự bịa đặt tên bảng không khớp, naming convention khác nhau, gây ra mã nguồn không thể chạy được.

Giải pháp: Biến projectName và userStories[] từ Agent 1 được inject trực tiếp vào prompt của Agent 2. Chuỗi SQL DDL từ Agent 2 được inject vào prompt Agent 3 — đảm bảo Controller UserController.js gọi đúng bảng Users như đã khai báo trong SQL, không bao giờ sai lệch.

2.4 Cơ Chế JSON Schema — Dynamic Response Control

Thay vì nhận chuỗi text tự do từ AI, hệ thống sử dụng tính năng Structured Output của Gemini API (tham số responseSchema + responseMimeType: "application/json"). Điều này ép AI phải trả về đúng định dạng object/array JSON, cho phép JSON.parse() hoạt động ổn định. 

// Ví dụ: JSON Schema cho Agent 3 — yêu cầu nhả ra mảng file động
const backendSchema = {
  type: "OBJECT",
  properties: {
    controllers: { type: "ARRAY",
      items: { type: "OBJECT",
        properties: {
          filename: { type: "STRING" },  // VD: "UserController.js"
          code:     { type: "STRING" }   // Nội dung file JS đầy đủ
        }
      }
    },
    // Tương tự cho services[], routes[]
  },
  required: ["readme", "controllers", "services", "routes"]
};
