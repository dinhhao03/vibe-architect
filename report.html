Chương 1: Giới thiệu đề tài

1.1 Bối cảnh và động lực

Trong lĩnh vực phát triển phần mềm hiện đại, giai đoạn khởi tạo dự án (project scaffolding) luôn tiêu tốn đáng kể thời gian và công sức của lập trình viên. Các thao tác lặp đi lặp lại như thiết lập cấu trúc thư mục MVC, cấu hình cơ sở dữ liệu, viết logic CRUD và soạn tài liệu đặc tả yêu cầu (SRS) chiếm trung bình 20–30% thời gian phát triển ban đầu của một dự án MVP.

Song song đó, sự bùng nổ của các mô hình ngôn ngữ lớn (Large Language Models — LLM) đã mở ra hướng tiếp cận mới: sử dụng AI để tự động hóa quy trình scaffolding. Tuy nhiên, các giải pháp hiện tại như ChatGPT hay Copilot thường hoạt động theo mô hình "single-shot" — nhận toàn bộ yêu cầu và sinh code một lần — dẫn đến hiện tượng ảo giác (hallucination), thiếu nhất quán giữa schema và logic, và chất lượng đầu ra khó kiểm soát.

Nghiên cứu về Agentic Workflow từ các tập đoàn như Google, Microsoft, và Anthropic đã chứng minh rằng việc phân chia tác vụ phức tạp cho nhiều Agent chuyên biệt, mỗi Agent đảm nhiệm một vai trò cụ thể (phân tích, thiết kế, lập trình), giúp cải thiện đáng kể chất lượng đầu ra nhờ cơ chế kiểm tra chéo và truyền ngữ cảnh liên tục.

1.2 Mục tiêu dự án

Dự án Vibe-Architect được phát triển nhằm đạt được các mục tiêu sau:

  1. Tự động hóa quy trình scaffolding: Biến một câu mô tả ý tưởng thành dự án Backend MVC hoàn chỉnh (code, database schema, tài liệu) trong vài chục giây.
  2. Triển khai Agentic Workflow: Xây dựng pipeline ba Agent chuyên biệt (Analyst → Architect → Developer) với cơ chế truyền ngữ cảnh xuyên suốt.
  3. Đảm bảo độ tin cậy tuyệt đối: Thiết kế hệ thống Cascading LLM + Response Cache để đạt 100% uptime, đặc biệt phục vụ cho các buổi trình diễn quan trọng.
  4. Hỗ trợ đa nền tảng: Sinh code cho cả hai stack phổ biến: Node.js (Express) và Python (Flask).

1.3 Phạm vi

Hệ thống hoạt động hoàn toàn trên môi trường Local (localhost), nhận đầu vào là prompt ngôn ngữ tự nhiên và xuất ra file ZIP chứa mã nguồn Backend MVC, tài liệu SRS, biểu đồ ERD, và SQL Schema. Dự án tập trung vào phần Backend; phần Frontend của sản phẩm cuối (ứng dụng được sinh ra) nằm ngoài phạm vi.

2.2 Multi-Agent Orchestration Pipeline

Thay vì giao toàn bộ tác vụ cho một lời gọi AI duy nhất (dễ gây ra hallucination và mất nhất quán), hệ thống chia quy trình thành 3 Agent hoạt động tuần tự với cơ chế Context Locking:

Agent Vai trò Đầu vào Đầu ra
Agent 1
Business Analyst		 Phân tích ý tưởng, trích xuất nghiệp vụ Prompt từ người dùng Project Name, Overview, User Stories (JSON)
Agent 2
Data Architect		 Thiết kế schema chuẩn 3NF SRS từ Agent 1 SQL DDL (PostgreSQL) + ERD Mermaid
Agent 3
Backend Developer		 Viết mã nguồn theo cấu trúc MVC SRS + SQL Schema Controllers, Services, Routes, Config files
Context Locking: Tên dự án (projectName) và SQL Schema được truyền xuyên suốt từ Agent 1 đến Agent 3, đảm bảo tên Entity trong Controller khớp chính xác với tên bảng trong Database.

2.3 Cascading LLM Architecture

Để giải quyết vấn đề API hết hạn mức (quota) — rủi ro lớn nhất trong các buổi demo — hệ thống triển khai cơ chế 8-Tier Cascading. Khi một model gặp lỗi 429 (Rate Limit) hoặc 503 (Overloaded), hệ thống tự động chuyển sang model tiếp theo mà không làm gián đoạn pipeline: 

Tier 1 (PRO — Chất lượng cao nhất):
  ├── gemini-2.5-pro
  └── gemini-1.5-pro

Tier 2 (FLASH — Nhanh, ổn định):
  ├── gemini-2.5-flash
  ├── gemini-2.0-flash
  └── gemini-1.5-flash

Tier 3 (LITE — Nhẹ, tiết kiệm token):
  ├── gemini-2.5-flash-lite
  └── gemini-2.0-flash-lite

Tier 4 (LOCAL MOCK — Dữ liệu mẫu, luôn hoạt động):
  └── Hardcoded Production-Ready Templates
Hình 2.2 — Sơ đồ Cascading LLM 8 tầng

Ngoài ra, khi model trả về lỗi 404 (Not Found), hệ thống sẽ bỏ qua ngay lập tức (không retry) và chuyển sang model tiếp theo để tiết kiệm thời gian.

2.4 Response Cache & Multi-Key Rotation

Response Cache (MD5-based): Mỗi tổ hợp prompt + systemInstruction được băm thành mã MD5 và lưu kết quả thành công vào thư mục .cache/. Khi cùng prompt được gọi lại, hệ thống trả kết quả từ cache với độ trễ 0ms và không tiêu tốn API call.

Multi-Key Rotation: Hệ thống hỗ trợ cấu hình tối đa 10 API Key trong file .env (biến GEMINI_API_KEY, GEMINI_API_KEY_2, ..., GEMINI_API_KEY_10). Khi một key hết hạn mức, hệ thống tự động chuyển sang key tiếp theo và reset lại cascade model từ đầu.

4.2 Module OrchestratorController

Đây là bộ não trung tâm của hệ thống, chịu trách nhiệm điều phối pipeline 3 Agent theo thứ tự tuần tự. Controller sử dụng giao thức SSE để stream tiến trình xử lý xuống Frontend theo thời gian thực.

Luồng xử lý chính:

  1. Thiết lập kết nối SSE (Content-Type: text/event-stream).
  2. Gọi Agent 1 với responseSchema kiểu JSON để ép AI trả về đúng cấu trúc {projectName, projectOverview, userStories[]}.
  3. Truyền kết quả Agent 1 vào prompt của Agent 2 → nhận về {erd, sql}.
  4. Chạy Validation Layer: kiểm tra SQL có chứa CREATE TABLE và ERD có hợp lệ không. Nếu không → chèn fallback template.
  5. Gọi Agent 3 với toàn bộ ngữ cảnh (SRS + SQL) → nhận về mảng code động: controllers[], services[], routes[].
  6. Gửi dữ liệu đến ScaffoldFactory để đóng gói ZIP.
  7. Stream event complete kèm đường dẫn download về Frontend.

4.3 Module AIClient (Cascading & Cache)

Module này triển khai 3 cơ chế bảo vệ quan trọng:

a) Cascading Logic

// Thuật toán Cascading (pseudocode):
for each API_KEY in [key_1, key_2, ..., key_N]:
    for each MODEL in [2.5-pro, 1.5-pro, 2.5-flash, ...]:
        try:
            result = call_gemini(MODEL, prompt)
            save_to_cache(result)
            return result
        catch 429/503:
            retry(1 time) → if fail → next MODEL
        catch 404:
            skip immediately → next MODEL
    // Hết model cho key này → next KEY

// Hết tất cả key + model → return MOCK data

b) Response Cache

// Tạo cache key bằng MD5:
cacheKey = MD5(systemInstruction + "||" + prompt)

// Đọc cache:
if (file ".cache/{cacheKey}.json" exists):
    return cached_response    // 0ms, 0 API calls

// Ghi cache sau khi AI trả về thành công:
save ".cache/{cacheKey}.json" ← { _cachedAt, response }

c) Multi-Key Rotation

Hệ thống đọc tất cả biến môi trường có dạng GEMINI_API_KEY, GEMINI_API_KEY_2, ..., GEMINI_API_KEY_10. Khi key hiện tại hết quota, hệ thống khởi tạo instance GoogleGenAI mới với key tiếp theo và reset cascade từ model đầu tiên.

Chương 6: Kết luận và hướng phát triển

6.1 Kết luận

Dự án Vibe-Architect v2.1 đã đạt được toàn bộ mục tiêu đề ra: xây dựng thành công một công cụ AI Scaffolding Engine hoạt động ổn định, sinh mã nguồn Backend chất lượng cao theo chuẩn MVC, và đảm bảo tính sẵn sàng tuyệt đối cho mọi tình huống sử dụng.

Điểm nổi bật của dự án nằm ở việc áp dụng thành công mô hình Agentic Workflow — một xu hướng tiên tiến nhất trong lĩnh vực AI Engineering hiện nay — vào bài toán scaffolding thực tế. Cơ chế Cascading LLM kết hợp Response Cache là giải pháp sáng tạo, đảm bảo hệ thống không bao giờ thất bại ngay cả khi toàn bộ API bị quá tải.

6.2 Hướng phát triển

  1. Bổ sung thêm Tech Stack: Mở rộng hỗ trợ Django (Python), Spring Boot (Java), và Laravel (PHP).
  2. Custom Template System: Cho phép người dùng upload template riêng (.ejs, .hbs) để AI sinh code theo mẫu.
  3. Multi-Entity Scaling: Nâng cấp prompt engineering để xử lý dự án lớn (> 5 entities) bằng kỹ thuật phân mảnh context.
  4. Unit Test Generation: Thêm Agent thứ 4 chuyên viết unit test cho các Controller được sinh ra.
  5. Docker Integration: Tự động thêm Dockerfile và docker-compose.yml vào ZIP output.
Tóm lại: Vibe-Architect v2.1 không chỉ là một bài tập học thuật mà là một công cụ phát triển phần mềm thực thụ, thể hiện năng lực vận dụng AI Agentic Workflow vào giải quyết bài toán tự động hóa trong kỹ thuật phần mềm hiện đại.
